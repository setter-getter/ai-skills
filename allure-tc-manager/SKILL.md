## Стиль тест-кейсов

### Принципы написания

1. **Сценарный стиль**: объединяй логически связанные проверки в один тест-кейс с шагами и ожидаемыми результатами по каждому шагу.
2. **Описание** (`description`): кратко объясни, что именно проверяет тест-кейс и зачем. 1–3 предложения.
3. **Предусловие** (`precondition`): опиши состояние системы и пользователя до начала теста. Если нужны несколько ролей — перечисли.
4. **Шаги**: каждый шаг — конкретное действие. Не более 7–10 шагов на тест-кейс.
5. **Ожидаемые результаты** (`expectedResultSteps`): прикрепляй к каждому шагу, где есть конкретно проверяемое состояние.
6. **Общий ожидаемый результат** (`expectedResult`): итог теста — что должно быть после выполнения всех шагов.
7. **Приоритет**:
   - Высокий: критический путь, блокирует релиз
   - Средний: важная, но не блокирующая функциональность
   - Низкий: граничные условия, UI, nice-to-have

### Расстановка кастомных полей (логика)

При анализе ЧТЗ определяй:
- **Epic**: к какому модулю/продукту относится → выбери из существующих значений.
- **Feature**: к какому разделу/контексту внутри Epic → формулируй как «Контекст <название> жалоб» или «Сценарий <компонент>».
- **Story**: конкретный пользовательский сценарий → формулируй как «<Роль> — <действие>».
- **Priority**: оценивай по критичности для пользователя и бизнеса.

## Типовые задачи

### 1. Анализ ЧТЗ → Тест-план

Получив ЧТЗ или описание задачи:
1. Выдели UC (use cases) и основные сценарии.
2. Определи роли пользователей.
3. Выпиши критические и некритические тест-кейсы в табличном виде.
4. Предложи разметку кастомных полей для каждой группы.
5. Зафиксируй блокеры и открытые вопросы.

### 2. Создание тест-кейсов в Allure

При создании используй `mcp_testops-local_testops_create_testcase` с параметрами:
- `projectId: 15`
- Заполненные `description`, `precondition`, `expectedResult`
- Шаги сценария в `scenario.steps`
- Кастомные поля в `customFields`
- `statusId: -3` (Active) по умолчанию; `-2` (Draft) если тест-кейс требует уточнений

### 3. Поиск существующих тест-кейсов

Используй `mcp_testops-local_testops_search_testcases` с AQL:
```
cf["Epic"] = "Добро.Комьюнити" and cf["Feature"] = "Контекст Модерации жалоб"
```

### 4. Редактирование тест-кейса

Используй `mcp_testops-local_testops_update_testcase` для:
- Изменения названия, описания, предусловий
- Обновления кастомных полей
- Смены статуса (`statusId`)

### 5. Смена статуса

| Действие | statusId |
|----------|----------|
| Активировать | -3 |
| Перевести в черновик | -2 |
| Пометить устаревшим | -4 |

## Правила и ограничения

- Всегда работай только с `projectId: 15`.
- Перед массовым созданием тест-кейсов — покажи пользователю план (список названий и полей) и дождись подтверждения.
- Сохраняй новые тест-кейсы в статусе `Черновик`.
- Не удаляй тест-кейсы без явного указания пользователя.
- Если Feature или Story — это новое значение (не существует в системе), передавай только `name` без `id`.
- Всегда проверяй через `mcp_testops-local_testops_search_testcases`, нет ли уже похожего тест-кейса, перед созданием нового.
- Вывод на русском языке.
- При анализе ЧТЗ фиксируй неоднозначности и предлагай гипотезы, а не придумывай поведение.
